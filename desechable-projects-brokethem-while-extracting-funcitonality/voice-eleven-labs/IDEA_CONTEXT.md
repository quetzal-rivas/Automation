It allows you to coordinate multiple "Named Agents" (IDE-Home, IDE-Work, GitHub-Action-CI) through a single centralized "Brain."To make this work across different environments, you need to implement Agent Identity and a State Machine (rather than just a simple queue).The "Global Directive Hub" Architecture1. The "Identity" LayerEvery instance of your IDE or GitHub Action needs a unique AGENT_ID (e.g., vscode-macbook-pro or gh-action-repo-alpha).The Task: When the VS Code task triggers the curl to your Lambda, it includes this AGENT_ID.The Context: Lambda passes this ID to ElevenLabs. ElevenLabs then includes it in the metadata when it calls your webhook back.2. The Directive Store (DynamoDB)Instead of a simple queue, use a database table to track the State of every interaction.agent_idsession_idstatusdirectivetimestampvscode-homexyz-123AWAITING_CALLNULL10:00:05gh-action-uiabc-789COMPLETED"Change padding to 20px"09:55:20vscode-workdef-456USER_MISSED_CALLNULL10:15:00Should you have a Queue?Yes, but inside the Database. If you use a standard queue (like AWS SQS), the message is deleted once it's read. In an IDE environment, you often want the agent to be able to "re-read" the last directive if it crashes or if you need to ask a follow-up question.How the Middleware Handles the "No Answer" Flow:Trigger: Agent A calls Lambda $\rightarrow$ Lambda starts ElevenLabs call $\rightarrow$ Lambda marks Agent A as CALL_IN_PROGRESS.The Missed Call: If you don't pick up, ElevenLabs sends a webhook with status: "no-answer".Middleware Logic: Lambda updates the DB to USER_MISSED_CALL.Agent Loop: Agent A polls the Lambda: "Any news?" Lambda responds: "The user missed the call. Should I try again in 10 minutes or wait for you to trigger manually?"The "Multi-Agent" Loop Logic1. Outbound (Pushing to Human)Any agent, anywhere, hits your Lambda:POST /trigger-call { "agent_id": "IDE_1", "summary": "Fixing bug #42" }Lambda registers the session and initiates the ElevenLabs outbound call.2. Inbound (Storing the Directive)ElevenLabs hits your Lambda Webhook:POST /webhook { "agent_id": "IDE_1", "transcript": "Change the color to blue" }Lambda updates the DB for IDE_1 with the new instruction.3. The Retrieval (Pulling to Agent)The Agents (IDE or GitHub Action) use an MCP tool to poll the Lambda:GET /get-directive?agent_id=IDE_1The Agent only pulls the instructions meant for it.Why this is better for your "Monday Morning" WorkflowIf you have a GitHub Action that runs on Sunday night, it can "queue up" a directive for you.Sunday 11:00 PM: GitHub Action finishes tests, fails, and hits Lambda.Lambda: Sees it's late, stores status as PENDING_MORNING_CALL.Monday 8:00 AM: Lambda (triggered by a cron job) finally initiates the ElevenLabs call to your phone.You: Give the directive.Monday 8:05 AM: You open your IDE at work. The IDE Agent polls the Lambda, sees the directive you just gave on the phone, and starts coding.Recommended Tech Stack for the MiddlewareAWS Lambda: To handle the webhooks (Scalable and cheap).Amazon DynamoDB: To store the agent states/directives.API Gateway: To expose the URL with Bearer Token security.MCP Server (Local): Still runs in your IDE, but now its only job is to fetch data from your Lambda API instead of listening for local tunnels.One Final Hack: The "Voice ID"Since you have multiple agents, you can tell the ElevenLabs agent to identify which one is calling:"Hi!
